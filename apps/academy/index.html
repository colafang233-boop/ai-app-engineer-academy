你会写 CRUD，为什么还写不好 AI 应用？

以前写 Java 接口，规则都在代码里：同样的参数进来，通常就得到同样的结果。 接入大模型后，模型会自己“判断”，但它不会天然知道你公司的规则，也不会天然返回后端能直接使用的数据。 这节课不先背术语——你会亲手把一个不稳定的模型调用，改造成能进正式项目的 AI 接口。

模型按什么规则判断

02 · STRUCTURED OUTPUT

后端拿到什么字段

不要让后端解析一段散文。用 Zod 定义字段、类型和枚举，让输出先变成可校验的数据。

z.object({ ... })
03 · BUSINESS VALIDATION

格式正确还不够

JSON 合法不代表业务合理。置信度太低、订单号缺失或规则冲突时，都不能自动往下走。

validate(result)
04 · FAILURE STRATEGY

失败以后去哪里

正式系统必须明确：重试、返回 unknown、要求补充信息，还是进入人工审核。

retry | unknown | human

classify-email.ts 
const EmailIntent = z.object({
  intent: z.enum(["refund", "refund_query", "complaint", "unknown"]),
  confidence: z.number().min(0).max(1),
  reason: z.string()
});

const messages = buildMessages(
  companyRules,
  email
);

const raw = await model.invoke(messages);
const result = EmailIntent.parse(raw);

if (!isBusinessSafe(result)) {
  return retryOrSendToHuman(email);
}

return result;
关键点不是“模型能不能看懂邮件”，而是：后端有没有明确告诉它怎么判断、要求它怎么返回，并在结果不可靠时阻止自动流转。

TypeScript 检查你写的代码，Zod 检查模型真的返回了什么

Compile time vs runtime

接口或 type 不能让外部数据自动变真。模型、HTTP 和数据库返回值必须先当作 unknown，再做运行时校验。

编译期

type Intent = "refund" | "query";
